JamVini Project Documentation

Developer documentation for the JamVini hosting platform.

This HTML documentation explains the core engine, plugin lifecycle, hooks, registries, REST API, cron system, and current extension points. It is project documentation, not a customer user guide.

Core Engine

JamVini is built as a Laravel application with a plugin-first engine. Core provides bootstrapping, authentication, routing, registries, hooks, shared helpers, installer behavior, and system-level orchestration. Business capabilities live in plugins.

Core Responsibilities

  • Plugin discovery through App\Core\PluginManager
  • Admin/client authentication and permissions
  • Menu, dashboard, report, registrar, payment, provisioning, and setting registries
  • Installer, migrations, update flow, cron manager, and API token middleware
  • Theme and plugin view namespaces

Plugin Responsibilities

  • Declare a plugin.json manifest
  • Ship routes, hooks, views, controllers, models, migrations, and services
  • Register widgets, reports, registrars, payment gateways, cron tasks, or provisioning modules
  • Keep behavior inside the plugin directory wherever possible

Plugin Structure

A production JamVini plugin should be installable without editing core files. The plugin folder name may use StudlyCase, but the manifest slug is the stable identifier used by menus, permissions, and plugin state.

plugins/ExamplePlugin/
  plugin.json
  routes.php
  hooks.php
  migrations/
    0001_create_example_tables.php
  src/
    Controllers/
    Models/
    Services/
  views/
    admin/
    public/
    client/

Manifest Example

{
  "name": "Example Plugin",
  "slug": "example-plugin",
  "version": "1.0.0",
  "core": false,
  "dependencies": ["clients"],
  "menu": {
    "admin": {
      "icon": "puzzle",
      "label": "Example",
      "route": "admin.example.index",
      "position": 80,
      "section": "main"
    }
  },
  "permissions": ["example-plugin.read", "example-plugin.write", "example-plugin.manage"]
}

Plugin Lifecycle

The lifecycle is managed by PluginManager. System plugins are synced from manifests. Active plugins expose routes and hooks. Fresh installs run migrations for core plugins.

StepWhat HappensDeveloper Notes
ScanReads every plugin manifest under plugins/.Manifest JSON must be valid.
InstallCreates a row in plugins.Does not necessarily activate.
ActivateRuns plugin migrations, routes, hooks, and marks active.Migration files must be idempotent.
BootRegisters active plugin menus, permissions, reports, and hooks.Hook files should stay light.
DeactivateDisables plugin and clears registered menu/widgets/permissions/reports.System plugins cannot be deactivated through normal flow.

Registries

Registries are lightweight in-memory extension points. Plugins register capabilities during boot, then controllers and views read from registries to build UI and behavior.

MenuRegistry

Registers admin and client menu entries from plugin manifests.

MenuRegistry::registerAdmin('example', [
  'label' => 'Example',
  'route' => 'admin.example.index',
  'icon' => 'puzzle',
  'section' => 'main'
]);

DashboardRegistry

Registers dashboard widgets with size, color, column, and admin-customizable order.

DashboardRegistry::registerWidget('sales-pulse', fn () =>
  view('plugins.Example::admin.widgets.sales'),
  ['title' => 'Sales Pulse', 'size' => 'medium']
);

ReportRegistry

Registers report definitions and handlers for the Reports plugin.

ReportRegistry::register('sales.monthly', ReportHandler::class, [
  'label' => 'Monthly Sales',
  'category' => 'Sales',
  'filters' => ['date_range']
]);

PermissionRegistry

Registers plugin permission modules. Admin routes use admin.permission:{module},{level}.

Route::middleware(['auth:admin', 'admin.permission:example-plugin,auto'])
  ->prefix('admin/example')
  ->name('admin.example.')
  ->group(...);

RegistrarRegistry

Registers domain registrars and supported TLDs.

RegistrarRegistry::register('manual', [
  'name' => 'Manual Registrar',
  'tlds' => ['.com', '.tz'],
  'class' => ManualRegistrar::class
]);

Payment / Provisioning

PaymentGatewayRegistry exposes enabled gateways. ProvisioningModuleRegistry selects hosting modules by server type.

PaymentGatewayRegistry::register('selcom', SelcomGateway::class);
ProvisioningModuleRegistry::register(new CpanelModule());

Hooks

Hooks allow plugins to observe or modify system behavior. Action hooks are event-style. Filter hooks transform a value. Both support priorities; lower priority numbers run first.

Action Hooks

use App\Core\Hooks\Action;

Action::add('client.created', function ($client) {
    // Send onboarding email, sync CRM, or log event.
}, priority: 10);

Action::do('client.created', $client);

Filter Hooks

use App\Core\Hooks\Filter;

Filter::add('invoice.total', function ($total, $invoice) {
    return $total;
}, priority: 20);

$total = Filter::apply('invoice.total', $total, $invoice);

Conditional Actions

Action::first() returns the first non-null result. Action::until() stops and returns false when a callback returns false.

$result = Action::first('domain.before_register', $domain);

if (! Action::until('service.before_suspend', $service)) {
    return back()->with('error', 'Suspension blocked.');
}

Known Hook Points

  • client.created, client.updated, client.deleting, client.deleted
  • client.registration.validation_rules, client.registration.created, auth.register.extra_fields
  • frontend.homepage, shortcode.{name}, api.local_action
  • invoice.paid
  • support.ticket_created
  • custom_fields.created, custom_fields.updated, custom_fields.deleted, custom_fields.values_saved
  • reports.registered, reports.exported
  • plugins.booted
Best practice: keep hook callbacks small. Put business logic in services, and queue slow external calls where possible.

RESTful API

JamVini exposes a versioned API under /api/v1. Requests must include a valid token through Authorization: Bearer {token} or X-API-Token. API calls are logged to api_logs when the table exists.

Authentication

curl -H "Authorization: Bearer YOUR_TOKEN" \
  https://example.com/api/v1/clients
MethodEndpointPurposeInternal Action
GET/api/v1/clientsList latest clientsGetClients
POST/api/v1/clientsCreate a clientCreateClient
GET/api/v1/invoicesList latest invoicesGetInvoices
GET/api/v1/servicesList latest client servicesGetServices
GET/api/v1/domainsList latest domainsGetDomains
GET/api/v1/ticketsList latest support ticketsDirect query
POST/api/v1/localCall an internal API actionDynamic

Creating a Client

curl -X POST https://example.com/api/v1/clients \
  -H "Authorization: Bearer YOUR_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "email": "client@example.com",
    "first_name": "Asha",
    "last_name": "Mushi",
    "phone": "+255712345678",
    "company_name": "Asha Hosting"
  }'

Internal API

App\Core\Api\InternalApi provides WHMCS-like local actions for hooks and modules. Current actions include CreateClient, UpdateClient, GetClients, GetInvoices, GetServices, and GetDomains.

$result = app(InternalApi::class)->call('CreateClient', [
  'email' => 'client@example.com',
  'first_name' => 'Asha',
]);

Plugin API Actions

Plugins can add local API actions without editing core by listening to api.local_action. Return null when the action is not yours.

Filter::add('api.local_action', function ($result, string $action, array $params) {
    if ($action !== 'MyPlugin.DoThing') {
        return $result;
    }

    return ['success' => true, 'data' => ['done' => true]];
}, 10, 3);

Cron

The cron engine is managed by App\Core\CronManager. Tasks are registered in PHP, stored in memory for the request, and their run state is tracked in cache and settings. Admins can view, run, disable, and regenerate the web cron key from the Cron Manager screen.

Register a Task

use App\Core\CronManager;

CronManager::register('sync-domain-expiry', 'daily', function () {
    // Update domain expiry dates.
    return 'Domain expiry sync complete.';
}, [
    'description' => 'Sync domain expiry dates'
]);

Supported Schedules

  • everyMinute, everyFiveMinutes, everyFifteenMinutes
  • everyThirtyMinutes, hourly, daily
  • weekly, monthly

Web Cron

For shared hosting, JamVini exposes /cron?key={cron_key}. The key is generated in the admin Cron Manager. The server cron should call that URL every minute.

* * * * * curl -s 'https://example.com/cron?key=YOUR_KEY' > /dev/null 2>&1
Cron tasks should be idempotent. A task may be retried after a failure, and long-running external calls should be guarded carefully.

Custom Fields

The Custom Fields plugin stores reusable field definitions in custom_fields and values in custom_field_values. It currently integrates with client registration and admin client profiles, and is structured for future order, service, domain, and ticket fields.

Supported Types

Text, textarea, email, URL, number, date, select, radio, checkbox, and yes/no fields.

Visibility Controls

Fields can be active/inactive, required, visible to clients, shown on registration, and shown on the admin profile.

app(CustomFieldService::class)
    ->fields('client', ['registration' => true]);

app(CustomFieldService::class)
    ->sync($client, 'client', request('custom_fields', []));

Reports

The Reports plugin uses ReportRegistry so core and plugins can publish report definitions. Built-in reports cover billing, clients, sales, domains, services, support, and SEO analytics.

ReportRegistry::register('billing.revenue-summary', [Handler::class, 'handle'], [
  'label' => 'Revenue Summary',
  'category' => 'Billing',
  'icon' => 'dollar-sign',
  'filters' => ['date_range'],
  'exportable' => true
]);

Report handlers should return summary, columns, rows, and optionally chart.

Security Guidelines

  • Admin routes should use auth:admin and admin.permission.
  • API routes should use jamvini.api and avoid returning secrets.
  • Plugin migrations should use explicit index names for MySQL compatibility.
  • Do not store raw API tokens. JamVini stores token_hash with SHA-256.
  • Hooks should not trust external payloads. Validate before creating records.
  • Background jobs and cron tasks should avoid leaking credentials into activity logs.

Fresh Install Checklist

  • Installer runs core Laravel migrations first.
  • Core plugin manifests are synced before active plugin routes are loaded.
  • Installer runs migrations for core plugins after sorting by dependencies.
  • Custom Fields should appear under the System menu after install.
  • Reports should appear in the main admin menu when active.
  • Run System → Run Migrations after adding plugin files to an existing installation.
  • Clear view and route cache after changing Blade views, routes, plugin manifests, or theme files.
